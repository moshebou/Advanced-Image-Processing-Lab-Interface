Preface

Advanced image processing lab is an important step on the path of a student to become an expert in the field of image processing and computer vision.

The lab gives the student the opportunity to link his accumulated theoretical knowledge with actual practices using tools which are industry standard, such as matlab.

More advantages which the lab provides is the dealing with real life problems, the ability to identify problems which may occur in real application and the way to fix those problems.

The “Advanced image processing lab project” aims to bring all those advantages together, to a single program which will:

Give a single interactive interface to many programs and experiments

Allows the student to customize the experiment and interactively modify the result

For experiments with graduate behavior, give the student and graduate appearance

Familiarize the students with professional notations from the field of image processing

Link between the theoretical to the practice

The following chapters will describe the structure of the program, each experiment theoretical background experiment description, and experiment results.

The project was greatly relied on the lectures and articles of Prof. Leonid Yaroslavsky, and most of the theoretical background is taken from Prof. Yaroslavsky book: “Theoretical Foundations of Digital Imaging Using MATLAB”.

1         Advanced image processing lab project layout

1.1      Matlab GUIDE tool

The advanced image processing lab project is graphical user interface for all of advanced image processing lab experiment. It is built with matlab infrastructure and with matlab built-in tool ‘GUIDE’, the Open GUI Layout Editor tool for gui infrastructures.

Matlabs ‘GUIDE’ stores the GUI in two files:

·         A FIG-file, with extension .fig, that contains a complete description of the GUI layout and each UI component, such as push buttons, axes, panels, menus, and so on. The FIG-file is a binary file and which cannot be modified except by changing the layout in GUIDE. FIG-files are specializations of MAT-files.

·         A code file, with extension .m, that initially contains initialization code and templates for some callbacks that control GUI behavior. The code file contains the call back of the main FIG file UI components.

The code file structure is built of:

·         Comments - Displayed at the command line in response to the help command.

·         Initialization - GUIDE initialization tasks. Needed for proper operation of the code.

·         Opening function - Performs your initialization tasks before the GUI user has access to the GUI.

·         Output function - Returns outputs to the MATLAB command line after the opening function returns control and before control returns to the command line.

·         Component and figure callbacks - Control the behavior of the GUI figure and of individual components. MATLAB software calls a callback in response to a particular event for a component or for the figure itself.

·         Utility/helper functions - Perform miscellaneous functions not directly associated with an event for the figure or a component.

All communication and data sharing between the different components are done with the ‘guidata’ function, which Store or retrieve GUI data.  Once a program, function or callback are finished and needs to save data for later use. For example, to save an experiment inputs for other run of the experiment, or UIs which may need to be deleted in later time.

The execution of the program is done by either of the following options:

1.        ‘cd’ to the folder which contains the two files (.fig and .m) , and run the program.

2.       Add to matlab path the folder which contains the program, and run the program.

3.      Run the command ‘run(full_path_of_program\program_name)’ where full_path_of_program is the full path that contains the program, and program_name is the program name.

A normal operation of a program with GUI starts with the execution of the main code file which was generated by ‘GUIDE’. This execution will first run the Initialization code and Opening function, each of which needs to save relevant data via guidata command.

Finally, the Output function run which determines the output (usually the handle of the figure). At that point the figure is already opened, and ready to be used. Once all the function has finished their tasks, the figure UI waits for the user interaction, an according to the specific design, callbacks and other functions are used.

 

1.2      Advanced image processing lab project

1.2.1     Advanced image processing lab project file structure

As described above, the advanced image processing program is built from 2 head files ‘AdvancedImageProcessingLab_mb.m’ and ‘AdvancedImageProcessingLab_mb.fig’, both were created using matlab guide tool. In addition, the program contains:

Twelve folders (one for each topic) – in order to clarify each topic, the files are separated into different folders according to the relevance. Only experiments of the same topic are in the same folder. Each folder contains the experiments and algorithms files relevant to that topic.

Input data folder – contains images and signals to be used in the program. The existence of this folder is necessary, since each experiment have a default image\signal residing in the ‘data’ folder.

‘utils’ folder -  contains common function and data base files. This folder added to the Matlab path when the program is opened.

‘htm’ folder - contains all the documentation. The documentation are word documents saved in an html format.

1.2.1.1     Execution of the program

When executing AdvancedImageProcessingLab_mb the Opening function performs the following tasks and saves it to GUI data:

·         Check the validity of key folders

·         Saves current Matlab status

·         Load default program parameters

·         Adds the ‘utils’ folder to Matlab path

·         Load 2 database files: TopiclList.mat and Defaults.mat, both stored at ‘utils’ folder.

o   TopiclList.mat contains all the task and experiments names, and is used for navigation between the different experiments.

o   Defaults.mat defines the default input signal\images in each experiment.

After the Opening function finished, Output function defines the output of the program (the figure handler), the main window appears and the UIs are ready to be used.

1.4      Tools bar

The tool bar contains tool which helps the user to inquire and investigate each experiment results. There is a callback function relating to each of those buttons in the ‘AdvancedImageProcessingLab_mb.m’.

The toolbar consists of 9 buttons:

Data cursor () – enables inquiries of pixel or plot values at a certain positions.

Load image\ load auxiliary image \ load 2nd auxiliary image () – allows the user to change the input signal\image used by current experiment. For experiments with more than one input, load auxiliary image \ load 2nd auxiliary image buttons should be used. All experiments have default input image \signal, however it is possible to use user’s images \ signals. All image formats supported by matlab “imread” function aer supported -   ( 'bmp', 'cur’, ‘fts’, ‘fits’, ‘gif’, ‘hdf’, ‘ico’, ‘j2c’, ‘j2k’, ‘jp2’, ‘jpf’, ‘jpx’, ‘jpg’, ‘jpeg’, ‘pbm’, ‘pcx',    'pgm’, ‘png’, ‘pnm’, ‘ppm’, ‘ras’, ‘tif’, ‘tiff’, ‘xwd’), and raw ‘mat’ gray scale image files, for which the user will be prompt to provide the image size. It is possible to load an image of any size, but each image is resized to the size of 256x256. If the image is a color image, it will transform to gray. If the aspect ratio is not 1x1, the center of the image is cropped, to avoid distorting the image.

Zoom in\out ()  - allows the user to inspect image\signal by zooming in or out.

Pan () – allows the user to shift the image/signal right, left up or down.

Help () – upon pressing this button while in experiment, a detailed help document will be opened, with experiment tasks, questions, theoretical background and algorithm descriptions. Upon pressing this button while in main window, this chapter will appear.

Interactive\non interactive mode () – This button controls the behavior of the program, when user changes the experiment setting. If it is set to ‘off’, then only once the user presses “run experiment’ will the experiment will be executed. If it is set to ‘on’, then whenever the user changes the experiment settings, immediately, the experiment is executed.

1.6      Topic window

Upon pressing a topic button, the pushbutton_Callback located at ‘AdvancedImageProcessingLab_mb.m’ is called, and according to the button string (the topic name) finds the relevant folder, cd to it, clears the main window, loads the experiment list to allow navigation between topic’s experiments,  and the “Back” button which returns to the main window.

1.7      Experiment program

Each experiment is built form three sections:

·         Experiment interface - the main function, which has the name of the experiment, defines the GUI interface of the experiment (axes, sliders, and buttons), sets the default parameters of the experiment, loads the image\signal if necessary, and place a “Run experiment” button.

Once the main function finished its run, its returned value is saved to the GUI data.

·         “process_image” function, which is for execution of algorithms, and displaying of the results. Usually runs only once the user presses the “Run experiment” button.

·         Other specific per experiment functions for setting the interface and running the algorithms

An example of standard experiment function structure is:

% An experiment function which called by SetParam, the callbacks of the sliders. Used to update sliders when there are inter-relations between parameters (for example external radius and internal radius.exist in all experiments.

function update_sliders(handles)

    if ( ~isstruct(handles))

        handles = guidata(handles);

    end

    if ( strcmpi(handles.interactive, 'on'))

        run_process_image(handles);

    end

    guidata(handles.figure1,handles );

end

 

% A function which takes the handle structure, and change it to specific parameters, to execute the experiment. Exist in all experiments.

function run_process_image(handles)

    if ( ~isstruct(handles))

        handles = guidata(handles);

    end

    LocalAdaptiveFiltering2D = handles.(handles.current_experiment_name);

    process_image(LocalAdaptiveFiltering2D.im, LocalAdaptiveFiltering2D.WndSz, LocalAdaptiveFiltering2D.std, ...

    LocalAdaptiveFiltering2D.p, LocalAdaptiveFiltering2D.g, LocalAdaptiveFiltering2D.Thr ,LocalAdaptiveFiltering2D.filter_type,  ...

    LocalAdaptiveFiltering2D.axes_1, LocalAdaptiveFiltering2D.axes_2, LocalAdaptiveFiltering2D.axes_3, LocalAdaptiveFiltering2D.axes_4);

end

 

 

 

% The heart of the experiment, the function which execute the algorithms and displays the result. In some experiments, some of results are not dependet of any of the parameters, and its execution is done at the run_process_image function

function     process_image(im, WndSz, std, p, g, Thr , filter_type, axes_1, axes_2, axes_3, axes_4)

        imshow(im,[0, 255], 'parent', axes_1);

        DisplayAxesTitle( axes_1, ['Input Image'], 'TM'); 

        im  = double(im);

        h = waitbar(0, 'please wait');

        noise = std*(randn(size(im)));

        im = im + noise;

        sp_mask = ones(WndSz, WndSz);

        imshow(im,[0, 255], 'parent', axes_2);

        DisplayAxesTitle( axes_2, ['Input Image With Noise'], 'TM'); 

        if ( strcmpi(filter_type, 'Rejecting filter'))

            k = 1 ;

        else

            k = 0;

        end

        [OUTIMG1, OUTIMG2]=lcdct2_mb(im,sp_mask,p,g,Thr, k, h, axes_3, axes_4);

        imshow(OUTIMG1,[0, 255],  'parent', axes_3);

        DisplayAxesTitle( axes_3, ['Cleaned Result'], 'BM'); 

       

        imshow(OUTIMG2, [0, 255] ,'parent', axes_4);

        DisplayAxesTitle( axes_4, ['DCT-Domain Mask (Controlled By Threshold]'], 'BM'); 

        waitbar(1,h);

        close(h);

end

1.8      Common functions

As it can be seen from the code example, there are several general functions which repeat themselves through the program.

This is a review of some of them:

1.8.1.1     DeployAxes

function  handles = DeployAxes( parent, axes_num, bottom, left, width, height, is_outerposition )

The DeployAxes function calculates the location of each axes position, and create those axes.

The parameter list of this function:

·         parent – the figure handle

·         axes_num – an array of cell which describes the layout of the axes. For example, if axes_num = [2, 2] the result will be 2 by 2 axes.

·         bottom – lowest point for the location of axes

·         left – left most point for the location of axes

·         width – fraction of the width available to use

·         height - fraction of the height available to use

·         is_outerposition (optional) – a binary vector the length of number of axes requested, which defines in the axes is smaller to give space to titles (if is 1) or not.

1.8.1.2     SetSliderParams

function interface_params = SetSliderParams( title, max_val, min_val, curr_val, jump_step, call_back, in_interface_params, k)

The SetSliderParams function defines a structure which describes a slider for a parameter. This structure is the passed to the SetInteractiveInterface function, which sets the sliders and other UIs.

The parameter list of this function:

·         Title – slider title

·         max_val – the maximum value

·         min_val – the minimum value

·         curr_val – the current value of the parameter

·         jump_step – the resolution of the parameter

·         call_back – the callback function which executes when the slider changes

·         in_interface_params – the structure which needs to be updated

·         k – the current number of the parameter 

It returns the same structure as in_interface_params, with one more element  in the structure array.

 

1.8.1.3     SetInteractiveInterface

function buttongroup_handle = SetInteractiveInterface(handles,children_params)

This function creates the control panel, and all the UI (sliders, pushbuttons, radio buttons).

The parameter list of this function:

·         handles – the handles of the GUI interface

·         children_params – A struct containing the description of each UI.

It returns a button group handler, to all of the UIs.

 

1.9       Program modification

1.9.1     Change default parameters of an experiment

Each experiment have a set of default values such as window size, thresholds etc.

The default values are defined at the beginning of a function with same name as the experiment (without spaces and special characters such as ‘-‘), in a file with same name as the experiment (without spaces and special characters such as ‘-‘).

In addition to the default parameters values, there are others properties which defined by default, such as parameters range, step size etc.

In order to change an experiment default parameter, one should simply change the code line corresponds to that parameter. For example: LocalAdaptiveFiltering2D.Thr = 5;

In order to change this parameter range, as described by the slider in the control panel, or to change the resolution (step size), one simply change the SetSliderParams function call for that parameter.

Important: in some experiments, the control panel changes according to the selection of experiment parameters. For example, in experiment “Localization Accuracy and Reliability” under the “Target Location” topic, the noise type determines the appearance of the control panel (for correlated noise 2 more sliders which determines the internal and external radiuses are added). This done in the callback function of the radio button which controls the noise.

In such cases, where a parameter controls the appearance of the control panel, a change done to the control panel in the main function should also be implemented in the definition of the control panel in the callback function.

1.9.2     Change default images\signals of an experiment

The default image\signal list is stored in the Defaults.mat file located in the ‘utils’ folder.

Defaults.mat contains a structure with 49 fields (one for each experiment). Each of the field name is the name of an experiment, where special characters and spaces are replaced by a ‘_’ character.

Each field contains up-to 3 fields: im_1, im_2, im_3. Most only one – im_1, some have also im_2, and one experiment have the im_3 field. The field im_x is a struct which contains 6 fields: filename, height, width, index, type and title.

·         filename – the file name with respect to the ‘utils’ folder,  i.e., for the lenna image in the data folder, file name should be ‘../data/lenna_256x256.img’

·         height, width – the height and width of the image. Relevant for raw image formats files with no header (img files)

·         index – internal parameter. Can be any number

·         type – ‘signal’ or ‘image’

·         title – the image title

1.9.3     Add/remove an experiment

In order to add/remove an experiment to an existing topic, the following steps should be completed:

To add an experiment:

1.      Create a Matlab function which implement the experiment in a file structure similar to the code example in Code ‎1‑1. File name and function name should be the same, and composed of a single word, with a ‘_mb’ suffix. The file should be located in the folder corresponding to the topic which the experiment belongs to.

2.      Define a default test image\signal to the experiment as described at ‎1.9.1. Add the relevant fields in the default structure.

3.      Add the experiment name to the experiment list at the TopicList.mat file, as described in the following chapter.

To remove an experiment:

1.      Delete the Matlab function which implement the experiment.

2.      Remove the experiment field in the ‘default’ struct saved in ‘Defaults.mat’ data base file under the ‘utils’ folder.

3.      Remove the experiment name from the experiment list at the TopicList.mat file, as described in ‎1.9.3.1.

 

1.9.3.1     Updating the ‘TopicList.mat’ file

The TopicList.mat file is a Matlab data base file for the program which located in the ‘utils’ folder, and describe the list of experiments for each topic. The TopicList.mat file contains a struct variable called ‘TopicList’ with 12 fields (one for each topic). Each field name is a topic name where special characters and spaces are replaced by a ‘_’ character. The fields contains a list of strings cells, one for each experiment of the topic, which contains the name of the experiment, with spaces and special characters.

An example of such struct can be seen at Figure ‎1‑6.

1.9.4      Add\Remove a topic

In order to add/remove a topic, the following steps should be completed:

To add a topic:

1.      Add a pushbutton to the GUI using Matlab ‘guide’ tool (preferable), or by direct coding in the main function (‘AdvancedImageProcessingLab_mb.m’). The handle to the pushbutton should be held in the handles of the GUI, under ‘pushbutton_x’ field name, where x is the pushbutton number. This is done automatically when using the ‘guide’ Matlab tool for creating the button. If using a direct coding, then it should be added to the opening function. For example, if ‘pb_handle’ is the pushbutton handle, then the line ’handles.pushbutton_13 = pb_handle’ should be added to the AdvancedImageProcessingLab_mb_OpeningFcn. The pushbutton string should be the topic name.

2.      Add the topic and its experiments to the ‘TopicList’ structure described at ‎1.9.3.1. in the following way:

a.       Change current directory to the ‘utils’ folder:

o   cd './utils/'

b.      Load the ‘TopicList’ structure:

o   load('TopicList.mat');

c.       Add the experiment names to a new field with name of the topic (without spaces and special characters) :

o    TopicList.NewTopic{1} = ‘New Topic Experiment1’;

o    TopicList.NewTopic{2} = ‘New Topic Experiment2’;

o    .......

d.      Save the struct in the ‘utils’ folder:

o   save('TopicList.mat', 'TopicList');

3.      Create a folder with the topic name, which will contains all the experiments files and experiments related files.

To remove a topic:

1.      (Necessary) Remove the pushbutton of the topic from the GUI using Matlab ‘guide’ tool. 

2.      (optional) Remove fileld from the ‘TopicList’ structure described at ‎1.9.3.1. in the following way:

a.       Change current directory to the ‘utils’ folder:

o   cd './utils/'

b.      Load the ‘TopicList’ structure:

o   load('TopicList.mat');

1.10  Algorithmic implementation

The Matlab program language, while very versatile and convenient, is aimed at vector and matrix processing, and has a major drawback in the form of execution time of loops. Loops over an image happened mainly when performing local operation.

In order make the advanced image processing lab project faster and interactive, there is a need to transform many of the loop implementation into matrix operation. This can be exemplified by the local histogram operation.

The standard local histogram operation would have to, for each pixel in the image, calculate the histogram of the window surrounding it. Experiment shows it can takes up to several seconds. A different approach can solve this issue buy the use of im2col function in maltab.

Im2col(im, [WndY, WndX]) function takes an image with size WxH, and creates a new matrix with size of  (WndY* WndX)x(W*H). Which means that for 5x5 window, this function creates 24 more replicas of the image. But this cost in memory benefits in execution time. Empirical result shows improvement of over 50 times.

In conclusion, many local image operation were implemented using im2col operation, however, in some cases for memory and educational purposes, a loop implementation was used.
